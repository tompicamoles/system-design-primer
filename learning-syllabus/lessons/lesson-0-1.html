LESSON 0.1

What is system design & why scalability matters

⏱ 45m Source: System Design Primer Module 0 · Foundations

Most software works fine for ten users. The interesting engineering begins when ten becomes ten million — when the same correct code starts to fall over, not because it's wrong, but because there's too much of everything: too many requests, too much data, too many things that can fail at once. System design is the discipline of arranging servers, databases, caches, and networks so a system keeps performing as load grows. This first lesson builds the vocabulary the rest of the course rests on.

By the end you'll be able to:
  • Define scalability precisely, and tell a performance problem apart from a scalability problem.
  • Explain vertical vs horizontal scaling and the trade-off between them.
  • Name the recurring levers — clones, caching, load balancing, replication, partitioning — that every later module zooms into.

🧠 Mental model

Think of a popular restaurant. One brilliant chef (your single server) can cook beautifully — that's performance. But when 300 people show up at once, the food doesn't get worse; it gets slow. You now have a scalability problem, and you have two ways to fix it:

  • Scale up — give the one chef a bigger kitchen, more burners, sharper knives. Powerful, but there's a ceiling, it's expensive, and if that chef calls in sick the restaurant closes.
  • Scale out — hire more chefs working identical stations, with a host (a load balancer) seating diners across them. Cheaper per chef, survives one chef quitting — but now the chefs must share recipes and not step on each other.

That single image — one big worker vs many identical small ones coordinated by a dispatcher — is the seed of nearly every system design decision you'll make.

📈 Scalability, defined

A service is scalable if adding resources increases performance proportionally to the resources added. The cleanest way to separate the two ideas people constantly confuse:

  • A performance problem → your system is slow for a single user.
  • A scalability problem → your system is fast for a single user but slow under heavy load.

They have different fixes. A performance problem is usually solved by better code or a faster algorithm. A scalability problem is solved by the architectural moves in this course.

⚙️ Two ways to scale

Vertical scaling (scale up)

Buy a bigger box — more CPU, RAM, and disk on one server. Simple, no code changes. But specialized high-end hardware is expensive, there's a hard physical ceiling, that one machine is a single point of failure, and it's harder to hire talent for exotic enterprise systems.

Horizontal scaling (scale out)

Add more commodity machines and spread load across them with a load balancer. More cost-efficient and more available than scaling up — but it introduces real complexity:

  • Servers must be stateless — they can't hold user data like sessions or profile pictures locally, because the next request may land on a different server.
  • Shared state (sessions) moves to a centralized store — a database or a persistent cache like Redis/Memcached.
  • Downstream resources (caches, databases) must now handle many more simultaneous connections as the server count grows.

🧭 The recurring levers (the map of this course)

The classic scalability lecture introduces a handful of levers you reach for again and again. Almost every later module is a deep dive into one of them:

  • Clones — identical stateless app servers behind the load balancer (Module 2–3).
  • Load balancing — spreading requests across the clones (Module 2).
  • Caching — keep hot results in fast memory instead of recomputing them (Module 5).
  • Database replication — copies of data for read scaling and availability (Module 4).
  • Database partitioning — splitting data so no single node holds it all (Module 4).
  • Asynchronism — push slow work off the request path onto queues (Module 6).

⚖️ Trade-offs

Everything is a trade-off. This is the single sentence to carry through the entire course. Horizontal scaling buys availability and cost-efficiency at the price of complexity and statelessness. There is no free win — only choices whose costs you accept on purpose.
👍 Why prefer horizontal scaling?
Cheaper commodity hardware, no hard ceiling (keep adding nodes), higher availability (one node dying isn't fatal), and an easier hiring market than for specialized enterprise gear.
👎 What does it cost you?
Architectural complexity: servers must be stateless, sessions move to a shared store, and downstream caches/DBs must absorb far more concurrent connections. You've traded simplicity for scale.

✅ Check yourself

Predict the answer out loud before opening each one — the act of retrieval is what moves this into long-term memory.

Your app loads instantly for you at 2am but crawls during the lunch rush. Performance or scalability problem?
Scalability. It's fast for a single user but slow under heavy load. A performance problem would be slow even for one user with no one else around.
Why must horizontally-scaled app servers be stateless?
Because the load balancer may route a given user's requests to a different server each time. If a server stored the session locally, the next request landing elsewhere wouldn't find it. So shared state lives in a centralized store (DB or a cache like Redis).
Name two costs of vertical scaling that horizontal scaling avoids.
(1) A hard physical ceiling on how big one machine can get, and (2) a single point of failure — that one box going down takes everything with it. (Also: expensive specialized hardware.)
